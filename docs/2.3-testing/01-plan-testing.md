# ПЛАН ФУНКЦИОНАЛЬНОГО ТЕСТИРОВАНИЯ
## Приложение "НеПросрочь!" для контроля сроков годности продуктов

---

## 1 ВВЕДЕНИЕ

### 1.1 Цель документа

Настоящий документ определяет стратегию, объем и методы функционального тестирования мобильного приложения "НеПросрочь!" для платформы Android.

### 1.2 Область применения

План тестирования применяется ко всем функциональным компонентам приложения "НеПросрочь!" версии 1.0.

## 2 ОБЪЕКТ ТЕСТИРОВАНИЯ

### 2.1 Общие сведения

**Наименование системы:** Мобильное приложение "НеПросрочь!"
**Платформа:** Android
**Версия:** 1.0
**Минимальная версия Android:** API 29 (Android 10)
**Язык разработки:** Kotlin
**Архитектура:** MVVM (Model-View-ViewModel)

### 2.2 Основные функциональные модули

1. **Модуль управления продуктами**
   - Добавление продукта
   - Редактирование продукта
   - Удаление продукта
   - Просмотр списка продуктов

2. **Модуль сканирования штрихкодов**
   - Распознавание штрихкодов с помощью камеры
   - Интеграция с Open Food Facts API
   - Автоматическое заполнение данных о продукте

3. **Модуль уведомлений**
   - Уведомления за 7 дней до истечения срока
   - Уведомления за 3 дня до истечения срока
   - Уведомления за 24 часа до истечения срока

4. **Модуль списка покупок**
   - Добавление товара в список
   - Отметка товара как купленного
   - Удаление товара из списка

5. **Модуль авторизации**
   - Регистрация пользователя (Email/Password)
   - Вход в систему
   - Выход из системы

6. **Модуль визуальной индикации**
   - Зеленая индикация (более 7 дней до истечения)
   - Желтая индикация (3-7 дней до истечения)
   - Красная индикация (менее 3 дней или просрочено)

## 3 СТРАТЕГИЯ ТЕСТИРОВАНИЯ

### 3.1 Уровни тестирования

| Уровень | Описание | Инструменты |
|---------|----------|-------------|
| Модульное тестирование | Тестирование отдельных функций и методов | JUnit 4, Kotlin Test |
| Интеграционное тестирование | Тестирование взаимодействия компонентов | JUnit 4, Mockito |
| Системное тестирование | Тестирование приложения в целом | Espresso, UI Automator |
| Приемочное тестирование | Проверка соответствия требованиям | Ручное тестирование |

### 3.2 Виды тестирования

1. **Функциональное тестирование** - проверка соответствия функций заявленным требованиям
2. **Тестирование пользовательского интерфейса** - проверка удобства использования
3. **Тестирование производительности** - проверка быстродействия приложения
4. **Тестирование безопасности** - проверка защиты данных пользователей
5. **Регрессионное тестирование** - проверка работоспособности после изменений

### 3.3 Критерии начала тестирования

- Завершена разработка модуля
- Модуль успешно скомпилирован
- Подготовлены тестовые данные
- Настроено тестовое окружение

### 3.4 Критерии окончания тестирования

- Выполнены все запланированные тест-кейсы
- Критических дефектов не обнаружено
- Покрытие кода тестами составляет не менее 70%
- Все найденные критические и высокие дефекты исправлены

## 5 МЕТРИКИ КАЧЕСТВА

### 5.1 Метрики покрытия

| Метрика | Целевое значение | Критическое значение |
|---------|------------------|----------------------|
| Покрытие требований тестами | 100% | 90% |
| Покрытие кода тестами | 80% | 70% |
| Покрытие критических путей | 100% | 100% |

### 5.2 Метрики дефектов

| Метрика | Описание |
|---------|----------|
| Критический дефект | Приложение неработоспособно или происходит потеря данных |
| Высокий дефект | Основная функция не работает, есть обходной путь |
| Средний дефект | Незначительная функция не работает |
| Низкий дефект | Косметические проблемы, не влияющие на функциональность |

### 5.3 Целевые показатели качества

| Показатель | Значение |
|------------|----------|
| Критические дефекты на момент релиза | 0 |
| Высокие дефекты на момент релиза | 0 |
| Средние дефекты на момент релиза | ≤ 3 |
| Низкие дефекты на момент релиза | ≤ 10 |
